# Analysis Lenses

Use these lenses to create independent candidate families. Add a domain-specific lens only when the brief clearly needs one.

## Unification Lens

Look for two algorithms, proof styles, or performance identities that appear different but may share a hidden object. Candidate output should name the common object and what theorem would make the unification nontrivial.

## Assumption-Removal Lens

Find a standard assumption that may be an artifact of the proof. Candidate output should state the weaker assumption, the old proof bottleneck, and the new proof lever.

## Converse or Lower-Bound Lens

Ask whether an existing positive result is tight. Candidate output should specify the adversarial construction, information-theoretic reduction, or counterexample family.

## Stability Lens

Turn an algorithmic update into a stability, contraction, Lyapunov, monotonicity, or error-propagation claim. Candidate output should name the metric or potential.

## Decomposition Lens

Split a hard performance gap into interpretable pieces. Candidate output should identify the decomposition identity and which term exposes the new insight.

## Bridge-to-Practice Lens

Use theory to explain a phenomenon observed in deep or approximate methods. Candidate output should stay formal: define the simplified model, prove the mechanism, then list only optional empirical checks.

## Failure-Mode Lens

Search for a clean counterexample to a common intuition. Candidate output should define the smallest setting where the intuition fails and why the failure is informative.
